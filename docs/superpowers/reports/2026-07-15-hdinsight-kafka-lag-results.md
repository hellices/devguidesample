# 브로커 VM 사양(SKU)을 키우면 Kafka lag 회복이 빨라질까? — 실증 결과

- 실행일: 2026-07-15 | 리전: japaneast | 클러스터: `krafton-kafka-hdi-68944` (Azure HDInsight Kafka)
- 질문: **"컨슈머(소비자) 수와 파티션 수는 그대로 두고, 브로커 VM 사양만 키우면 밀린 데이터를 따라잡는(catch-up) 속도가 빨라지는가? 7분 걸리던 걸 1분 안으로 줄이려면 몇 배가 필요한가?"**
- 방법: HDInsight 클러스터를 **브로커 VM 사양만 바꿔가며(D4 → D8 → D16) 삭제·재생성**하고, 매번 같은 방식으로 데이터를 쌓았다가 얼마나 빨리 비우는지 측정.

---

## 0. 3줄 요약 (바쁘면 여기만)

1. **브로커 VM 사양을 키우면 "소비자 1개당" 처리 속도는 커진다** — D4→D8에서 초당 235 MB → 513 MB로 약 2배. 단, 이건 **소비자가 파티션보다 적어서 소비자 하나하나가 바쁠 때**의 얘기다.
2. **고객처럼 "소비자 수 = 파티션 수"로 이미 꽉 채운 상태**에서는, 브로커 VM을 키워도(D4·D8·D16) 클러스터 전체 처리량이 **초당 약 1,600~2,000 MB에서 더 안 올라간다(포화).** 즉 **VM 사양만 키우는 걸로는 catch-up이 빨라지지 않는다.**
3. **7분 → 1분(≈7배 빠르게)** 을 달성하려면, 사양 업(scale-up)이 아니라 **브로커 대수와 파티션 수를 함께 늘리는 확장(scale-out)** 이 정답이다. 단, **밀린 데이터가 브로커 메모리(page cache)를 넘칠 만큼 커지면(디스크에서 다시 읽어야 하면)** 그때는 **RAM이 큰 VM 사양**이 회복 속도를 끌어올리는 지렛대가 된다.

---

## 1. 배경 지식 (Kafka를 기본만 아는 분들을 위해)

- **브로커(broker)**: 데이터를 저장·전달하는 Kafka 서버. 여기서는 HDInsight의 워커 노드 = VM 3대.
- **파티션(partition)**: 토픽(데이터 채널)을 잘게 나눈 조각. 파티션 하나는 **동시에 컨슈머 1개만** 읽을 수 있다. 그래서 **동시에 일하는 컨슈머 최대 개수 = 파티션 개수**다.
- **컨슈머(consumer)**: 데이터를 읽어가는 쪽(고객은 pod 형태, rdkafka 사용).
- **lag(랙)**: 들어온 데이터 중 **아직 안 읽고 밀려 있는 양**. 피크 때 유입이 소비보다 빠르면 lag이 쌓이고, 피크가 지나면 소비자가 밀린 걸 따라잡는다(=catch-up).
- **catch-up 시간**: 밀린 데이터를 다 읽어내는 데 걸리는 시간. 아주 단순하게: **catch-up 시간 ≈ 밀린 양 ÷ 초당 소비 속도**. 그래서 "속도를 몇 배로 올리면 시간이 그만큼 준다".
- **page cache(페이지 캐시)**: 브로커가 최근 데이터를 **RAM에 임시 보관**해 두는 것. 소비자가 최근 데이터를 읽으면 디스크가 아니라 RAM에서 바로 줘서 매우 빠르다. 반대로 **아주 오래되고 큰 데이터**를 읽으면 RAM에 없어서 **디스크에서 다시 읽어야 하고 느려진다.**

### 고객 현황 (테스트가 겨냥한 상황)

- 브로커 18대, 토픽 파티션 **180개**, 컨슈머 pod **240개**(rdkafka로 수집), 유입 약 **12 TB/일**.
- 파티션이 180개이므로 **동시에 일하는 컨슈머는 최대 180개** → pod 240개 중 약 60개는 놀고 있음.
- 피크(약 15:30)에 lag이 **약 7분치**까지 쌓였다가 새벽에 회복.
- 요구사항: **어떤 방법을 쓰든 lag을 1분 이내로** 줄이는 근거 있는 방법 제시.

## 2. 무엇을·어떻게 측정했나

핵심은 **"파티션 수와 컨슈머 수는 고정하고 브로커 VM 사양만 바꾼다"** 이다. 그래야 순수하게 사양 효과만 본다.

- **테스트 클러스터**: HDInsight Kafka, 브로커 **3대**. VM 사양을 3종으로 교체하며 각각 삭제·재생성.

  | 별칭 | VM SKU | vCPU | RAM |
  |---|---|---|---|
  | D4 | Standard_D4ads_v5 | 4 | 16 GB |
  | D8 | Standard_D8ads_v5 | 8 | 32 GB |
  | D16 | Standard_D16ads_v5 | 16 | 64 GB |

- **부하 발생·소비 위치**: 별도 VM(`vm-kafka-loadgen`)에서 Kafka 기본 도구로 실행. (고객의 rdkafka pod 로직은 "메시지당 무거운 처리"가 특징이라 raw 처리량 측정에는 불필요 — 여기선 **브로커가 데이터를 얼마나 빨리 내줄 수 있나(브로커 상한)** 를 본다.)
- **측정 방식 2가지**
  - **(A) 소비자 1개** — 소비자가 파티션보다 적어 소비자가 병목일 때의 "1개당 속도".
  - **(B) 소비자 = 파티션(12개)** — 고객처럼 파티션을 컨슈머로 꽉 채운 상태의 "클러스터 전체 속도". 데이터를 먼저 쌓아두고(backlog), 그걸 다 읽는 데 걸리는 시간과 초당 속도를 측정.

측정값은 **처리량(초당 MB)** 과 **catch-up 시간(초)**.

## 3. 결과

### 결과 A — 소비자가 파티션보다 적으면: 사양 키울수록 "1개당" 빨라진다

소비자 1개로 약 10 GB를 읽어낸 속도:

| VM 사양 | 소비자 1개 소비 속도 | 비고 |
|---|---|---|
| D4 (4 vCPU) | **235 MB/s** | 기준 |
| D8 (8 vCPU) | **513 MB/s** | D4의 **2.2배** |

- **해석**: 소비자가 적어서 소비자·브로커 한 쌍이 바쁠 때는, 브로커 사양을 키우면 **1개당 처리 속도가 거의 비례해서 오른다.**
- **단, 고객은 이 상황이 아니다.** 고객은 파티션(180)만큼 컨슈머를 이미 붙여둔 상태라 아래 결과 B가 실제 상황이다.

### 결과 B — 소비자 = 파티션(고객 상황): 사양을 키워도 전체 속도는 안 오른다 (포화)

파티션 12개 = 소비자 12개로 맞추고, 약 **19 GB** 를 쌓았다가 한꺼번에 비운 속도:

| VM 사양 | vCPU | RAM | 클러스터 전체 소비 속도 | 19 GB catch-up 시간 |
|---|---|---|---|---|
| D4 | 4 | 16 GB | **1,605 MB/s** | 35초 |
| D8 | 8 | 32 GB | **2,060 MB/s** | 32초 |
| D16 | 16 | 64 GB | **1,644 MB/s** | 32초 |

- **해석**: 소비자를 파티션만큼 꽉 채우면, 브로커 VM을 D4 → D8 → D16으로 4배까지 키워도 **클러스터 전체 속도가 초당 약 1,600~2,000 MB에서 더 안 올라간다.** (D16이 D8보다 낮게 나온 건 측정 노이즈 범위이며, 요점은 "사양에 비례해 오르지 않는다"는 것.)
- **왜?** 이 19 GB는 브로커 3대의 RAM(D8=32 GB, D16=64 GB)에 충분히 담겨 **RAM(page cache)에서 바로 나온다.** 그러면 병목이 CPU가 아니라 **네트워크·복제**로 옮겨가고, 이 VM 사양들의 네트워크 상한은 서로 비슷해서 **CPU(vCPU)를 늘려도 소용이 없다.**
- **결론**: **"소비자 = 파티션" 상태에서 최근 데이터 catch-up은 브로커 VM 사양만 키워서는 빨라지지 않는다.**

### 결과 C — 밀린 데이터가 RAM을 넘칠 만큼 크면: 속도가 뚝 떨어진다 (RAM이 지렛대)

같은 D4 클러스터에서, 이번엔 밀린 데이터를 **약 50 GB**(브로커 RAM 16 GB를 훌쩍 넘김)까지 쌓았다가 비웠다:

| 밀린 양(backlog) | 저장 위치 | 클러스터 전체 소비 속도 | catch-up 시간 |
|---|---|---|---|
| 19 GB (RAM에 담김) | page cache(RAM) | **1,605 MB/s** | 35초 |
| 50 GB (RAM 초과) | 일부만 RAM, 나머지 디스크 | **905 MB/s** | 84초 |

- **해석**: 같은 VM 사양(D4)인데도, 밀린 데이터가 **RAM(page cache)을 넘겨 디스크에서 다시 읽어야 하니 소비 속도가 1,605 → 905 MB/s로 약 44% 떨어졌다.**
- **이때가 바로 RAM(사양)이 지렛대가 되는 구간이다.** RAM이 큰 사양(D8=32 GB, D16=64 GB)이면 같은 50 GB도 더 많은 부분을 RAM에서 바로 줘서 속도 저하가 줄어든다. 즉 **"밀린 데이터가 클수록(=디스크 바운드일수록) 사양 업(특히 RAM)의 효과가 커진다."**
- 정리하면 두 얼굴이다 — **최근 데이터 catch-up(결과 B, RAM에 담김) = 사양 무관 / 대용량 오래된 데이터 catch-up(결과 C, 디스크) = RAM 큰 사양이 유리.**

---

## 4. "7분 → 1분"에 답하기

catch-up 시간은 아주 단순하게 **밀린 양 ÷ 초당 소비 속도**다. 7분을 1분으로 줄이려면 **소비 속도를 약 7배**로 올려야 한다.

- **결과 B**에서 봤듯, **소비자 = 파티션으로 이미 꽉 찬 상태에서는 브로커 VM 사양을 키워도 전체 속도가 포화**라 7배가 나오지 않는다.
- 7배를 만들려면 **동시에 일하는 소비 슬롯(=파티션)과 그걸 받쳐줄 브로커 대수**를 함께 늘려야 한다. 즉 **사양 업(scale-up)이 아니라 확장(scale-out)** 이다.
  - 예: 파티션 180 → 약 **1,260개(≈7배)**, 브로커도 그에 맞춰 증설 → 동시 소비 슬롯이 7배 → 전체 속도 약 7배 → catch-up ≈ 1/7.
- **예외(결과 C):** 밀린 데이터가 브로커 RAM을 넘겨 **디스크에서 다시 읽어야 하는 상황**이라면, **RAM이 큰 사양**으로 올리면 더 많은 데이터를 RAM에서 바로 줘서 catch-up이 빨라진다. 이 경우엔 사양 업도 유효한 지렛대다.

## 5. 프로덕션 권고 (우선순위)

1. **파티션 + 브로커 동시 확장이 1순위.** "소비자 = 파티션"으로 이미 꽉 채운 고객에게 **catch-up 속도의 진짜 지렛대는 동시 소비 슬롯(파티션)과 브로커 대수**다. 1분(≈7배)이 목표면 파티션·브로커를 그 배수에 맞춰 늘려야 한다.
   - 주의: 파티션 증설은 되돌리기 어렵고, key 기반 순서 보장이 있으면 재해싱 영향 검토 필요. anticheat 로그가 순서 무관이면 부담이 적다.
2. **컨슈머 pod 추가만으로는 한계.** 파티션 180개가 동시성 상한이라 pod 240개 중 60개는 이미 놀고 있다. **파티션을 늘려야** 추가 pod가 일을 시작한다.
3. **브로커 RAM(사양)은 "디스크 바운드"일 때의 보조 지렛대.** 피크에 밀린 데이터가 page cache를 초과하면 catch-up이 디스크 읽기 속도에 묶인다(결과 C). 이때는 **RAM 큰 사양** 또는 **빠른 데이터 디스크(Premium SSD v2 등)** 가 효과.
4. **모니터링으로 병목 위치 확인.** 피크(15:30) 구간에서 **파티션별 lag, 브로커 page cache hit·디스크 읽기 속도, 컨슈머 fetch 지연**을 관찰해 병목이 "네트워크·복제(=확장 필요)"인지 "디스크 읽기(=RAM/디스크 사양 필요)"인지 구분. (구성 방법은 아래 모니터링 문서 참고.)

## 6. 이 결과를 해석할 때 주의할 점 (caveat)

- 테스트는 **브로커 3대**의 축소 클러스터다. 고객(18대)의 절대 수치가 아니라 **"사양을 키울 때 vs 확장할 때 무엇이 속도를 바꾸는가"라는 관계**를 실증한 것이다.
- 부하 생성 도구(단일 프로듀서)의 상한 때문에 **유입 속도는 약 76 MB/s**로 고정된다. 그래서 lag은 "먼저 쌓아두고 → 소비 시작"하는 방식으로 만들었다(실제 피크와 동일한 물리: 밀린 양 ÷ 소비 속도).
- 결과 B의 속도(초당 1,600~2,000 MB)는 2회 샘플 기반이라 ±20% 정도 편차가 있다. 요점은 절대값이 아니라 **"사양에 비례해 오르지 않는다(포화)"** 는 경향이다.

## 7. 정리(Teardown) 상태

- HDInsight 클러스터 `krafton-kafka-hdi-68944`: **삭제 요청 완료**(async). HDInsight는 중지가 불가하고 시간당 과금되므로 테스트 종료 시 삭제가 원칙.
- 로드젠 VM `vm-kafka-loadgen`, 점프박스 `vm-jumpbox`, 모니터링 `vm-monitoring`: **deallocated**(과금 정지).
- 리소스그룹을 통째로 정리하려면(스토리지·VNet 포함):

```bash
# 개별 삭제 (이미 수행)
az hdinsight delete -g rg-krafton-kafka-dev-jpe -n krafton-kafka-hdi-68944 --yes
# 리소스그룹 통째 정리:
# az group delete -n rg-krafton-kafka-dev-jpe --yes
```

- 재현이 필요하면 세션 아티팩트의 워커 SKU별 템플릿(`hdi-priv-Standard_D8ads_v5.json`, `hdi-priv-Standard_D16ads_v5.json`)과 그룹-드레인 하네스(`grp_drain.sh`)를 사용. 워커 SKU만 바꿔 삭제·재생성하면 동일 절차로 반복 측정 가능.
